# White Paper: The Snowflake Oracle Protocol
**Decentralized Time-Lock Encryption via Threshold Consensus**

## 1. Introduction
Current time-lock systems rely on "trusted" servers to hold a key until a timer expires. If the server is compromised or the administrator is coerced, the data is leaked. The **Snowflake Oracle Protocol** solves this by ensuring that no single node holds the decryption key. Instead, the key is a mathematical product of the entire P2P mesh, generated only when a verifiable consensus of nodes agrees that a specific **Snowflake ID** (time) has passed.

## 2. Core Concepts
### 2.1 Snowflake IDs as Entropy Anchors
In Gorgona, every message and time-event is represented by a 64-bit Snowflake ID. Because Snowflake IDs contain a high-precision timestamp, they serve as the "Identity" for the encryption. To encrypt a message for the future is to encrypt it for a specific **future Snowflake ID**.

### 2.2 Pairing-Based Cryptography (BLS)
The protocol utilizes **Boneh-Lynn-Shacham (BLS)** signatures and **Identity-Based Encryption (IBE)**. This allows a sender to encrypt data using a public string (the future Snowflake ID) as the public key. The corresponding private key can only be generated by combining "partial signatures" from a quorum ($K$ of $N$) of nodes.

### 2.3 Distributed Key Generation (DKG)
During the mesh bootstrap, nodes perform a decentralized setup.
- **Global Public Key ($P_{pub}$):** Known by all nodes and clients.
- **Node Secret Share ($s_i$):** A private fragment held by each node, never shared.
- **Threshold ($K$):** The minimum number of nodes required to "witness" time.

## 3. The Operational Flow

### Phase I: Encryption (Sender)
1. **Target Identification:** The sender defines $T_{future}$ and calculates the corresponding `SnowflakeID_Target`.
2. **Double Wrapping:**
   - **Layer 1:** Standard RSA encryption (Recipient's Public Key).
   - **Layer 2 (The Vault):** AES encryption using a key $K_{time}$.
3. **Identity-Based Locking:** The $K_{time}$ is not stored. Instead, it is encrypted using the **Global Public Key ($P_{pub}$)** and the string `SnowflakeID_Target`.
4. **Replication:** The doubly-encrypted payload is replicated across all nodes in the mesh (High Availability).

### Phase II: The Embargo (Storage)
While $Now < T_{future}$, the payload is mathematically undecryptable. Even if an attacker seizes the recipients RSA key and the raw database from multiple nodes, the $K_{time}$ does not exist yet. It cannot be derived until the nodes produce their partial signatures.

### Phase III: Time-Consensus Verification (P2P)
When a client requests the message at $T_{future}$:
1. **The Request:** The node receives a request for the time-unlock share for `SnowflakeID_Target`.
2. **Local Verification:** The node checks its own `MaxID`. If `MaxID < SnowflakeID_Target`, it refuses.
3. **Mesh Cross-Check:** The node queries $X$ random peers: *"Confirm your current MaxID"*.
4. **Attestation:** If the quorum confirms the time, the node uses its **Node Secret Share ($s_i$)** to produce a **Partial Signature ($\sigma_i$)** for that specific Snowflake ID.

### Phase IV: Reconstruction (Recipient)
1. **Collection:** The recipient collects $\sigma_i$ from any $K$ nodes in the mesh.
2. **Combination:** Using the BLS combination algorithm, the recipient merges the partial signatures to recreate the **Master Signature ($\Sigma$)**.
3. **Decryption:** The Master Signature $\Sigma$ is used as the $K_{time}$ to remove the AES layer, followed by RSA decryption.

## 4. Security & Resilience
- **Anti-Coercion:** A node operator cannot be forced to give up the key, as they only possess a fragment.
- **Time-Fudging Protection:** A compromised node that manually advances its system clock will fail the P2P cross-check, as neighboring nodes will refuse to sign the attestation.
- **High Availability:** Because the payload is replicated and the keys are threshold-based, the message can be recovered as long as *any* $K$ nodes remain alive in the mesh.

## 5. Implementation Roadmap (C11)
### 5.1 Cryptographic Library
- **MCL (Min-Curve Lib):** Recommended for C-friendly, high-performance pairing-based cryptography (BLS12-381 curve).
### 5.2 Protocol Extensions
- **New P2P Commands:**
  - `CMD_TIME_ATTEST`: Request a partial signature for a specific Snowflake ID.
  - `CMD_QUORUM_INFO`: Exchange information about the current DKG epoch and threshold.
### 5.3 Storage
- **History Log Idempotency:** Ensure the `history.log` tracks partial signature requests to prevent DDoS on the cryptographic engine.

---

**Summary:**
The **Snowflake Oracle Protocol** moves Gorgona from a *server-enforced* time-lock to a *mathematically-enforced* time-lock. It ensures that the only way to read a message from the future is for the P2P mesh to reach a collective consensus that the future has indeed arrived.
Which makes it possible for the data owner to access the data, even if they possess the keys, only once a valid future has been reached, as confirmed by the network’s quorum.
